# P9.5-PARALLEL-1 — Concurrent /ctl/ accept

**Status**: design-first (spec-first applies — TLA+ in `v2/specs/ctl-conn.tla`).
**Predecessor**: SWISS-4g (FS-socket concurrent accept; commit `9b51bd2` era).
**Workaround it retires**: SWISS-8k (`166056b`) — pollers drop /ctl/ between
ticks to avoid serial-accept starvation. Once this chunk lands, long-lived
pollers can hold their /ctl/ connections again.
**Audit class triggered**: stratumd row + /ctl/ row in CLAUDE.md.
**Spec**: `v2/specs/ctl_conn.tla` (this chunk introduces it).
**Spec composition**: extends doctrine in `v2/specs/fid.tla` (per-connection
fid table) and `v2/specs/namespace.tla` (per-connection state isolation).

## 1. Problem statement

Stratum v2 ships two listen sockets per daemon:

- **FS socket** — speaks 9P2000.L bound to `stm_fs *` via `stm_9p_server`.
  Has been **concurrent-accept since SWISS-4g**: every accepted client gets a
  detached pthread (`stratumd_fs_worker`); the FS thread of `stm_stratumd_run`
  spawns workers and never blocks on serve.
- **/ctl/ socket** — speaks 9P2000.L against the operator-state synfs
  (`stm_ctl`) via the generic `stm_lp9_server`. **Still serial-per-socket**:
  the /ctl/ worker thread runs `accept → serve_ctl_client → close → accept`
  one client at a time.

The asymmetry is observable. SWISS-8k's bug surfaced because:

1. The TUI's F2 view runs **two pollers** (`VolumeMapPoller` +
   `SnapshotGraphPoller`), each holding a long-lived `CtlClient` connection
   to `/ctl/`.
2. Whichever poller's `thread::spawn` wins the start-order race grabs the
   /ctl/ accept slot indefinitely.
3. The losing poller's `connect()` succeeds (kernel queues the SOCK_STREAM
   connect on the listen backlog) but its first 9P Tversion blocks
   forever — the serial worker is busy serving the winner.

The clientside workaround (`client = None;` at the end of every successful
tick) re-establishes a fresh connection each second so the workers
take turns. It works but costs the cache + a ~4 ms reconnect handshake per
tick per poller, and any future long-lived /ctl/ client (Prometheus
scraper, Grafana dashboard, Thylacine kernel daemon) inherits the same
constraint.

**Thylacine's kernel-9P mount path cannot tolerate this.** A kernel client
holds its 9P session open for the lifetime of the mount. If the kernel
mounts the FS socket AND the user-space TUI also wants /ctl/ visibility,
the TUI's connection becomes second-class.

P9.5-PARALLEL-1 lifts the /ctl/ accept loop onto the SWISS-4g pattern:
**pthread-per-connection, detached, one `stm_lp9_server` per worker**.

## 2. What's hard about concurrent /ctl/

The FS server (`stm_9p_server`) was already designed for concurrent use
(one server per connection; the fid table is per-server). The /ctl/ vops
backend (`stm_ctl`) wasn't. Three pieces of `stm_ctl` state are stamped
**per-connection** but live as **singletons** on the shared instance:

### 2.1 `c->caller_uid` / `c->caller_gid` — last-writer-wins

```c
struct stm_ctl {
    uid_t caller_uid;    /* set by stm_ctl_set_caller every accept */
    gid_t caller_gid;
    ...
};
```

Today's serial accept calls `stm_ctl_set_caller(ctl, peer_uid, peer_gid)`
**before** `stm_lp9_server_create`, then `stm_ctl_set_caller(ctl, -1, -1)`
**after** serve returns. The admin gate (`ctl_caller_is_admin`) reads
`c->caller_uid` during vops dispatch.

Under concurrent accept:

```
T1 (peer uid_X):  set_caller(uid_X) ... [reads c->caller_uid → uid_X] ...
T2 (peer uid_Y):                       set_caller(uid_Y) ... [reads → uid_Y]
T1:                                                                       ... [reads c->caller_uid → uid_Y]  ← WRONG
```

A non-admin client could read admin-only files because a concurrent
admin connection's `set_caller` clobbered the singular field. **Confused
deputy via cross-connection state aliasing.** P0.

### 2.2 `c->sessions[]` — keyed by fid alone, namespace collision

```c
ctl_session sessions[STM_CTL_MAX_SESSIONS];
/* session_get_locked walks sessions[] looking for active && fid == X */
```

The fid namespace is **per-server-connection** (every `stm_lp9_server` has
its own fid table). Two concurrent clients can independently allocate
fid=1, fid=2, etc. — there's no global fid space.

Today's `session_alloc_locked` searches the **global** sessions[] array
for `c->sessions[i].fid == fid`. Under concurrent regime, two clients
both opening fid=1 would race; mutex serializes the alloc, but the
matched slot has no notion of which server's fid=1 it belongs to. Later
reads from either connection would target the same slot — the wrong
materialised body, the wrong snapshot_len, the wrong `caller_uid`
shadow stamped onto the session.

Worse, when one client clunks fid=1, `session_free_locked` clears the
slot — orphaning the other client's session. STM_CTL_MAX_SESSIONS = 64
depletes permanently across connections.

P0 — silent data corruption (cross-client reads) + permanent
slot-pool depletion.

### 2.3 `stm_ctl_drop_all_sessions` — drops sibling-connection state

```c
void stm_ctl_drop_all_sessions(stm_ctl *c) {
    /* walks sessions[] and frees every active slot */
}
```

Called between sequential connections (R101 P1-1 doctrine). Under
concurrent accept, calling drop_all when connection A finishes would
free connection B's still-active sessions. P0 — UAF on B's next vops
dispatch.

### 2.4 Subsystem pointers (`c->fs`, `c->pool`, `c->scrub`) — already-safe

The fs/pool/scrub pointers are **set at startup before any worker
thread spawns** (R97 P2-2 timing rule). After spawn, they're read-only.
The pthread_create call provides a happens-before barrier. No change
needed under concurrent regime.

### 2.5 `c->admin_uid` — already-safe

Same posture as fs/pool/scrub — set once at startup, read-only after.

### 2.6 `c->event_buf` — already-protected by `c->mu`

Mutex-guarded today. Under concurrent regime, the mutex still serializes
log writes correctly. No change to event_buf semantics, but the mutex
**covers more than it needs to**: today it guards sessions[] **and**
event_buf. After the refactor, sessions[] move to per-conn so the
shared mutex can shrink to event_buf only (and we rename it
`event_mu` for clarity).

## 3. The split: stm_ctl + stm_ctl_conn

The shape that resolves §2.1–§2.3 cleanly:

```c
struct stm_ctl {                    /* PROCESS-WIDE SHARED STATE */
    struct stm_fs    *fs;
    struct stm_pool  *pool;
    struct stm_scrub *scrub;
    uid_t            admin_uid;

    pthread_mutex_t  event_mu;       /* guards event_buf */
    uint8_t         *event_buf;
    size_t           event_len, event_cap;
};

struct stm_ctl_conn {                /* PER-CONNECTION STATE */
    stm_ctl         *ctl;            /* back-pointer to shared */
    uid_t            caller_uid;
    gid_t            caller_gid;
    pthread_mutex_t  mu;              /* guards sessions[] */
    ctl_session      sessions[STM_CTL_MAX_SESSIONS];
};
```

### 3.1 New API surface

```c
/* Allocate a per-connection wrapper. Caller's responsibility: keep
 * `ctl` alive longer than every `stm_ctl_conn` that holds it. The
 * conn does NOT take ownership of ctl. caller_uid/gid stamp the
 * peer's identity for the lifetime of the conn (immutable).
 * Returns STM_OK / STM_EINVAL / STM_ENOMEM. */
STM_MUST_USE
stm_status stm_ctl_conn_create(stm_ctl *ctl,
                                uid_t caller_uid, gid_t caller_gid,
                                stm_ctl_conn **out);

/* Free a per-connection wrapper. Walks sessions[] freeing any still-
 * active slots (defensive — lp9_server_destroy already issued vops_clunk
 * for every open fid before this point). Safe on NULL. */
void stm_ctl_conn_destroy(stm_ctl_conn *cn);

/* Optional accessor for the connection's peer uid (audit log + tests). */
uid_t stm_ctl_conn_caller_uid(const stm_ctl_conn *cn);
```

### 3.2 What changes on the existing API

`stm_ctl_set_caller`: deprecated (becomes a stub that errors with
`STM_ENOSYS`). Per-conn caller is set at conn-create time and immutable
thereafter. The function stays in the header with a deprecation note
through the next minor; the implementation refuses calls so any caller
that hasn't migrated breaks loudly rather than silently aliasing.

`stm_ctl_drop_all_sessions`: deprecated (becomes a no-op). Per-conn
sessions die with the conn; the "drop_all between connections" idiom is
obsolete. Same loud-deprecation posture.

`stm_ctl_vops`: signature stays. The vops contract now says **ctx is
`stm_ctl_conn *`**, not `stm_ctl *`. This is a doc-only change at the
vops type — `void *` allows the swap without ABI breakage.

`stm_ctl_log_event`: signature stays (logs are process-wide). Internally
uses `c->event_mu` (was `c->mu`).

### 3.3 vops migration

Every `vops_*` function in `synfs.c`:

```c
/* Before: */
static stm_status vops_lopen(void *ctx, uint32_t fid, ...) {
    stm_ctl *c = (stm_ctl *)ctx;
    pthread_mutex_lock(&c->mu);
    ctl_session *s = session_alloc_locked(c, fid, qid_path);
    ...
    if (KIND_META[k].admin_required && !ctl_caller_is_admin(c))
        return STM_EACCES;
    ...
}

/* After: */
static stm_status vops_lopen(void *ctx, uint32_t fid, ...) {
    stm_ctl_conn *cn = (stm_ctl_conn *)ctx;
    pthread_mutex_lock(&cn->mu);
    ctl_session *s = session_alloc_locked_cn(cn, fid, qid_path);
    ...
    if (KIND_META[k].admin_required && !ctl_caller_is_admin_cn(cn))
        return STM_EACCES;
    ...
}

static bool ctl_caller_is_admin_cn(const stm_ctl_conn *cn) {
    if (cn->caller_uid == (uid_t)-1) return false;
    if (cn->caller_uid == 0) return true;
    if (cn->ctl->admin_uid != (uid_t)-1 &&
        cn->caller_uid == cn->ctl->admin_uid) return true;
    return false;
}
```

For materializers that read subsystem state (`materialize_state`,
`materialize_pool_status`, etc.): they switch from `c->fs`, `c->pool`,
`c->scrub` to `cn->ctl->fs`, `cn->ctl->pool`, `cn->ctl->scrub`. The
underlying subsystem accessors (`stm_fs_stats_get`, `stm_pool_lock_shared`,
`stm_scrub_status_get`) are already thread-safe.

The `stm_ctl_log_event` calls inside materializers (e.g.
`materialize_clear_events`'s post-write log) take `cn->ctl` as the log
target.

### 3.4 stratumd wiring

The /ctl/ accept loop mirrors the FS pattern:

```c
typedef struct {
    int       client_fd;
    stm_ctl  *ctl;
    uid_t     peer_uid;
    gid_t     peer_gid;
    uint32_t  msize_max;
    uint32_t  idle_timeout_ms;
} stratumd_ctl_worker_ctx;

static void *stratumd_ctl_worker(void *arg) {
    /* R113 P1-1: block fatal signals so they route to the main thread. */
    sigset_t block;
    sigemptyset(&block);
    sigaddset(&block, SIGINT);  sigaddset(&block, SIGTERM);
    sigaddset(&block, SIGHUP);  sigaddset(&block, SIGQUIT);
    (void)pthread_sigmask(SIG_BLOCK, &block, NULL);

    stratumd_ctl_worker_ctx *ctx = arg;
    (void)stm_stratumd_serve_ctl_client(ctx->client_fd, ctx->ctl,
                                          ctx->peer_uid, ctx->peer_gid,
                                          ctx->msize_max,
                                          ctx->idle_timeout_ms);
    free(ctx);
    return NULL;
}

stm_status stm_stratumd_accept_ctl_loop(...) {
    while (!stop) {
        ... poll-then-accept (unchanged) ...
        ... peer_creds (unchanged) ...

        stratumd_ctl_worker_ctx *ctx = malloc(sizeof *ctx);
        if (!ctx) { close(client_fd); continue; }
        ctx->client_fd = client_fd;
        ctx->ctl       = ctl;
        ctx->peer_uid  = peer_uid;
        ctx->peer_gid  = peer_gid;
        ctx->msize_max = msize_max;
        ctx->idle_timeout_ms = idle_timeout_ms;

        pthread_t tid;
        int wprc = pthread_create(&tid, NULL, stratumd_ctl_worker, ctx);
        if (wprc != 0) { close(client_fd); free(ctx); continue; }
        (void)pthread_detach(tid);
    }
    return STM_OK;
}
```

`stm_stratumd_serve_ctl_client` then becomes per-conn:

```c
stm_status stm_stratumd_serve_ctl_client(int fd, stm_ctl *ctl,
                                          uid_t peer_uid, gid_t peer_gid, ...) {
    ... peer-cred clamp + setsockopt unchanged ...

    stm_ctl_conn *cn = NULL;
    stm_status rc = stm_ctl_conn_create(ctl, peer_uid, peer_gid, &cn);
    if (rc != STM_OK) { close(fd); return rc; }

    stm_lp9_server *srv = NULL;
    rc = stm_lp9_server_create(stm_ctl_vops(), cn,    /* ← per-conn ctx */
                                  stm_ctl_root(ctl), msize_max, &srv);
    if (rc != STM_OK) { stm_ctl_conn_destroy(cn); close(fd); return rc; }

    ... per-message loop (unchanged) ...

    stm_lp9_server_destroy(srv);
    stm_ctl_conn_destroy(cn);  /* replaces drop_all_sessions */
    close(fd);
    return rc;
}
```

`stm_stratumd_set_caller` calls REMOVED from the per-connection
pre/post boilerplate — caller is now bound at conn-create.

### 3.5 Shutdown ordering

Unchanged in shape but more careful in detail. The accept-loop pthread
returns when `stop_flag` is observed. Detached workers may still be
serving when stop fires. Today's FS-side posture: workers run to
natural end (EOF/error); OS reclaims stacks at process exit. We adopt
the same posture for /ctl/.

The hazard: `stm_ctl` is destroyed in `stm_stratumd_run` **after** the
accept-loop thread joins. But a detached worker holding a
`stm_ctl_conn *` might still be inside a vops dispatch reading
`cn->ctl->pool` when `stm_ctl_destroy(ctl)` runs. UAF on the worker.

**Fix**: introduce a worker-count refcount on `stm_ctl` (or extend the
existing pattern). Each worker increments at create + decrements at
destroy; `stm_ctl_destroy` waits via a condvar for the count to drop
to zero before tearing down.

```c
struct stm_ctl {
    ...
    pthread_mutex_t worker_mu;
    pthread_cond_t  worker_cv;
    uint32_t        worker_count;   /* live conns */
};

stm_status stm_ctl_conn_create(...) {
    ...
    pthread_mutex_lock(&ctl->worker_mu);
    ctl->worker_count++;
    pthread_mutex_unlock(&ctl->worker_mu);
    ...
}

void stm_ctl_conn_destroy(stm_ctl_conn *cn) {
    ...free sessions...
    pthread_mutex_lock(&cn->ctl->worker_mu);
    cn->ctl->worker_count--;
    if (cn->ctl->worker_count == 0)
        pthread_cond_broadcast(&cn->ctl->worker_cv);
    pthread_mutex_unlock(&cn->ctl->worker_mu);
    free(cn);
}

void stm_ctl_destroy(stm_ctl *c) {
    pthread_mutex_lock(&c->worker_mu);
    while (c->worker_count > 0)
        pthread_cond_wait(&c->worker_cv, &c->worker_mu);
    pthread_mutex_unlock(&c->worker_mu);
    ...rest of cleanup unchanged...
}
```

stratumd shutdown order remains:

```
1. observe stop_flag, exit FS accept loop
2. shutdown(2) on ctl listen fd to break /ctl/ accept poll
3. pthread_join the /ctl/ accept-loop thread
4. /ctl/ accept thread has returned, so no new workers spawn
5. stm_ctl_destroy: waits for in-flight workers to drain
6. stm_scrub_close, stm_fs_unmount
```

Detached workers don't need `pthread_join` because they exit naturally
on client disconnect AND `stm_ctl_destroy` blocks until they all do.
This is strictly more lifecycle than today (today's FS side relies on
process exit; we now bound at daemon shutdown). The FS side could
benefit from the same refcount but that's a separate chunk.

### 3.6 Lock ordering

After the split:

- `stm_ctl_conn::mu` (per-conn, guards sessions[]) — fine-grained
- `stm_ctl::event_mu` (process-wide, guards event_buf) — coarse
- `stm_ctl::worker_mu` (process-wide, guards worker_count) — coarse,
  used only at conn create/destroy + at shutdown

**No path takes both `mu` and `event_mu`**: materializers that read
sessions[] don't write events; log_event doesn't read sessions[]. If a
future writable kind logs from inside a sessions[]-holding path, we'd
acquire in the order `mu → event_mu`. The single permitted order is
documented in `synfs.c` header.

**No path takes both `mu` and `worker_mu`**: worker_mu is only at
conn-create + conn-destroy + destroy. Conn-destroy frees sessions[]
**without** mu (the conn is going away; the per-conn mu is destroyed
inside conn-destroy, not held across the worker_count decrement).

### 3.7 R94 P2-1 stat-after-mutation under concurrent regime

The CLAUDE.md stratumd row says: "Reviewer of any concurrent-accept
upgrade MUST address the stat-after-mutation race class (R94 P2-1
carry)." This applies to the FS-side accept (already concurrent) **and
to /ctl/ now**.

For /ctl/: vops reads go through `stm_fs_stats_get`, `stm_pool_lock_shared`,
`stm_scrub_status_get`, `stm_snapshot_iter`. Each is documented as
thread-safe (takes its own lock). The concurrent /ctl/ regime adds N
readers against these — exactly the regime they're designed for.

The novel concern for /ctl/ specifically: **between Tgetattr and Tlopen
(or between Twalk and Tread), the backend state can change under us.**
A snapshot a client just probed may have been deleted by the moment
they Tlopen. CLAUDE.md /ctl/ row clause (12) calls this out: "Every
walk + lopen + getattr re-checks the snap_id exists against the current
snapshot index under the appropriate lock." This is a per-kind
invariant the materializers already uphold for snapshot/dataset reads.

We don't change this behaviour; under concurrent regime, each
connection's vops calls see whatever the subsystem returns at that
instant. STM_ENOENT mid-walk is the expected error.

R131 (the audit chunk for P9.5-PARALLEL-1) should verify each kind's
materialiser **explicitly** for stale-id re-checks under the
new regime.

## 4. What the TLA+ model captures

`v2/specs/ctl_conn.tla` models the per-connection state isolation
discipline. It does **not** model the full vops surface — it abstracts
the read/write/admin-gate down to actions that exercise the
state-isolation invariants.

**Modeled state**:

- `conns` — set of currently-live connection IDs
- `caller[c]` — uid bound to conn `c` (immutable post-create)
- `sessions[c]` — set of (fid, qid) pairs owned by conn `c`
- `adminUid` — process-wide admin uid (immutable post-init)

**Actions**:

- `AcceptConn(c, u)` — birth: pick a fresh conn id, stamp caller=u
- `OpenFid(c, f, q)` — vops_lopen: bind fid f in c's session table
- `ReadFid(c, f)` — vops_read: requires fid present in c's table
- `AdminProbe(c)` — vops_walk through admin dir: requires caller(c) is
  admin (uid 0 or adminUid)
- `Clunk(c, f)` — vops_clunk: drop fid f from c's table
- `DestroyConn(c)` — connection ends; sessions[c] cleared, c removed

**Invariants (good run)**:

- `FidNamespacePerConn` — for any two distinct live conns c1, c2 and
  any fid f, if (f, q1) ∈ sessions[c1] and (f, q2) ∈ sessions[c2] then
  q1 may equal q2 OR differ; the spec does NOT require uniqueness across
  conns (per-conn fid spaces are independent). The invariant we actually
  verify is the dual: **a Clunk(c1, f) never removes (f, _) from
  sessions[c2]**. This is `NoClunkSpillover`.
- `CallerScopedPerConn` — `caller[c]` is set exactly once at AcceptConn
  and never mutates until DestroyConn. AdminProbe(c) decides
  `is_admin(caller[c], adminUid)` — never `caller[c']` for c' ≠ c.
- `NoSessionLeakAcrossConn` — when DestroyConn(c) fires, sessions of
  OTHER conns are unchanged.
- `LifecycleNoUAF` — at any state where conns is non-empty, the shared
  state (adminUid, plus subsystem-pointer abstractions) is still alive;
  conversely, "destroy shared state" can only fire when conns = {}. This
  is the worker-count refcount discipline.

**Buggy configs**:

- `ctl_conn_shared_sessions_buggy.cfg` — alternate action `OpenFidGlobal`
  writes into a singleton sessions[] keyed by fid only. Trips
  `NoClunkSpillover`: Clunk(c1, f=1) wipes the entry that OpenFid(c2,
  f=1, q2) created.
- `ctl_conn_shared_caller_buggy.cfg` — alternate `SetCallerGlobal`
  action overwrites a singleton `caller` field; AdminProbe reads the
  singleton. Trips `CallerScopedPerConn` when caller[c1]=uid_X and a
  concurrent AcceptConn(c2, uid_Y) flips the singleton to uid_Y mid-flight.
- `ctl_conn_destroy_uaf_buggy.cfg` — `DestroyShared` fires while conns
  is non-empty. Trips `LifecycleNoUAF`.

**Out of scope (handled by code-level convention, not the spec)**:

- The internal mutex contract on per-conn `mu` and process-wide
  `event_mu`/`worker_mu`. The spec assumes per-conn state mutations
  are atomic; the C implementation upholds this via the per-conn mutex.
  The buggy configs don't model the mutex elision because the worst-case
  data race is dominated by the structural bugs above.
- Signal-mask discipline. The R113 P1-1 carry is in code review territory.
- The actual 9P wire framing. Lives in `v2/specs/fid.tla` already.

**Bounds for TLC** (in `ctl_conn.cfg`):

- `Conns = {c1, c2}`, `Fids = {f1}`, `Qids = {q_state, q_admin}`,
  `Uids = {0, 1, 2}`, `AdminUid = 2`, `MaxOpens = 1`. Yields a small
  reachable space (low tens of thousands of states); full check
  expected well under 30 s on a typical workstation. The two
  connections + one fid + two qids combination is the minimum that
  exercises the cross-conn-aliasing pattern; larger bounds would
  inflate state without changing the invariants the model catches.

**TLC verification** is task #958 (P9.5-PARALLEL-1c). TLC isn't
installed on this machine and the sandbox blocked the tla2tools.jar
download, so the spec lands without a local run. The expected
verdicts are written into each cfg's header comment so a reviewer
or future-self can confirm in one TLC invocation per config:

```
java -jar tla2tools.jar -config ctl_conn.cfg ctl_conn.tla            \\
  → green; ≥10K states; all 5 invariants hold.

java -jar tla2tools.jar -config ctl_conn_shared_sessions_buggy.cfg \
                       ctl_conn.tla
  → red; trace fires NoReadAliasing within ~10 states.

java -jar tla2tools.jar -config ctl_conn_shared_caller_buggy.cfg \
                       ctl_conn.tla
  → red; trace fires CallerScopedPerConn within ~15 states.

java -jar tla2tools.jar -config ctl_conn_destroy_uaf_buggy.cfg \
                       ctl_conn.tla
  → red; trace fires LifecycleNoUAF within ~5 states.
```

| 3 | Deprecate `stm_ctl_set_caller` + `stm_ctl_drop_all_sessions` to no-op stubs (return STM_OK but log a warning at most-once via `stm_ctl_log_event`). | `synfs.c`, `ctl.h` |

| 6 | Tests: `test_ctl_concurrent` exercises two `stm_ctl_conn` simultaneously against one `stm_ctl`; verifies per-conn isolation. | `tests/test_ctl_concurrent.c`, CMake |

Steps 1–3 are "data-only" refactor (no behaviour change for the serial

concurrent loop. Step 6/7 prove the new regime. Step 8 retires the
workaround. Step 9 documents. Step 10 audits.

## 6. Test strategy

### 6.1 Unit (C)

`tests/test_ctl_concurrent.c`:

1. Create one `stm_ctl` with `admin_uid = geteuid()`.
2. Spawn two pthreads. Thread A creates `stm_ctl_conn` with
   `caller_uid = geteuid()` (admin); thread B creates with
   `caller_uid = 65535` (non-admin).
3. Each thread runs a tight loop: open fid=1 against the /admin/peer
   kind, read body, clunk. 1000 iterations.
4. Assertions:
   - Thread A always succeeds (admin).
   - Thread B always gets STM_EACCES (non-admin).
   - No sessions[] slot exhaustion (both threads cleanly clunk).
   - No mutex deadlock.

The buggy build (revert the conn split, use singleton state) should fail
this test loudly within iterations 1–10.

`tests/test_ctl_conn_lifecycle.c`:

1. Create `stm_ctl`.
2. Create N=4 `stm_ctl_conn` in succession; verify `worker_count` reaches 4.
3. Destroy each conn; verify `worker_count` decrements.
4. Spawn a pthread that creates one more conn + sleeps 100 ms.
5. From main: call `stm_ctl_destroy(c)`. Verify it BLOCKS until the
   pthread destroys its conn.
6. Verify no UAF under valgrind/asan.

### 6.2 E2E (Rust, libstratum-9p)

`v2/tools/stratum/tests/concurrent_ctl.rs`:

1. Start stratumd with `--ctl-listen`.
2. Spawn two threads. Each dials /ctl/ with libstratum-9p, walks to
   `/pools/`, reads readdir, holds the connection open for 5 seconds
   reading `/state` every 100 ms.
3. Assertion: both threads complete all reads; neither blocks for more
   than 50 ms on any single read.

The pre-fix build would fail this assertion (thread B blocks indefinitely
on Tversion until thread A disconnects).

## 7. Risks + mitigations

| Risk | Mitigation |
|------|------------|
| Worker-count refcount becomes a bottleneck (every conn lifecycle hits it). | The mutex/cv is only on create + destroy, not on the hot dispatch path. O(1) per conn. |
| stm_ctl_destroy blocks at shutdown if a misbehaving client holds an idle conn. | idle_timeout_ms (default 30 s) bounds. After that the worker's read fails → conn destroyed → refcount drops. |
| Deprecated stm_ctl_set_caller / drop_all_sessions break out-of-tree callers. | None expected — these functions are stratumd-internal. The deprecation log will surface any unexpected user within one mount. |
| R94 P2-1 stat-after-mutation surfaces new bugs at /ctl/ read paths. | R131 audit explicitly looks for them. The /ctl/ read paths already use thread-safe subsystem accessors; the audit just verifies. |
| Concurrent /ctl/ + concurrent FS exposes shared `stm_fs` to even more parallel readers/writers. | Already-supported per `stm_fs` thread-safety doc; the FS side has been concurrent since SWISS-4g. |

## 8. What this chunk doesn't address

- **Per-inode fs->lock granularity** (P9.5-PARALLEL-3) — still uses one
  big `fs->lock`. Concurrent /ctl/ + concurrent FS will serialize on
  this. PARALLEL-3 addresses.
- **Compound-op race audit** (P9.5-PARALLEL-2) — handlers that do
  multi-step state changes (snapshot create flush+commit, recv side
  effects) need pre-flush insertions under concurrent regime. PARALLEL-2
  addresses.
- **POSIX surface gaps for kernel-9P** (P9.5-POLISH-1) — Linux v9fs
  needs things like Tlock byte-range locks and Tgetlock. Already partly
  modeled in `locks.tla`; the wire glue is pending.

These are scoped to follow-up chunks. P9.5-PARALLEL-1 is the
minimum-viable concurrent /ctl/.

## 10. Why now (Thylacine context)

The Thylacine OS uses Stratum as a backing store via the Linux v9fs
kernel-9P client. The kernel mount holds an FS-socket connection for
the duration of the mount; concurrent user-space tools (`stratum
tui`, `prometheus`, ad-hoc CLI scrapes) all want simultaneous /ctl/
visibility. Today's serial /ctl/ accept makes any two such tools
fight for the slot.

Without this chunk, the user-facing TUI degrades (the SWISS-8k
workaround buys correctness at the cost of every poller losing its
cache 1×/sec). With this chunk, the TUI behaves like any production
NMS dashboard against a real daemon.
