# 20 â€” 9P2000.L filesystem server (P9-9P-1..3)

## Purpose

Stratum's **filesystem-bound** 9P2000.L server. Every external client
of the filesystem â€” FUSE shim, CLI, libstratum-9p, future Linux
kernel v9fs mount, Stratum-TUI v2 via the FFI shim â€” comes through
this surface (ARCH Â§10.3 "9P-first stance"). The server owns a
non-owning reference to a `stm_fs *` and dispatches each wire op
directly to the corresponding `stm_fs_*` API.

Distinct from:

- `<stratum/p9.h>` (janus's vops-table 9P2000 codec for `/keys/`),
- `<stratum/lp9.h>` (generic vops-table 9P2000.L codec â€” what /ctl/
  uses).

This is the only one of the three that is FS-bound: there is no
vops indirection. Each Txx handler reads / writes the fs directly.

One server instance = one client connection = one fid namespace.
Cross-connection concurrency is the daemon's problem (`stratumd`
spawns a fresh server per accept); see `21-stratumd.md`.

The 9P2000.L dialect is defined by the diod project
(https://github.com/chaos/diod/blob/master/protocol.md) and used by
the Linux kernel's v9fs since 2.6.34.

Header: `v2/include/stratum/9p.h` (413 lines).
Impl: `v2/src/9p/server.c` (3702 lines).
Specs: `v2/specs/fid.tla` + `v2/specs/namespace.tla`.

### Stratum extensions (P9-9P-2 + P9-9P-3)

Numeric range 124â€“139 chosen past the 9P2000.L assigned ops:

```
Tbind / Tunbind    (124..127) â€” per-connection namespace mutation
Tsync               (128/129) â€” explicit whole-pool sync (Stratum-only)
```

The Tfallocate / Tfadvise flag constants are **numerically equal to
the Linux UAPI values** (drift-asserted via `_Static_assert` in
`server.c`) so a v9fs client can pass kernel `<linux/falloc.h>` and
`<linux/fadvise.h>` values unchanged.

### Errno-on-the-wire table

Rlerror's `ecode` is a Linux errno value (per spec). On Linux this
equals `<errno.h>`; on macOS / *BSD the host's `<errno.h>` differs,
so `status_to_errno()` routes through `STM_9P_ECODE_*` so a v9fs
client sees Linux values regardless of the daemon's host OS.

### msize ceiling

```
STM_9P_MSIZE_MIN       4 KiB
STM_9P_MSIZE_DEFAULT 128 KiB  (diod's default)
STM_9P_MSIZE_MAX       8 MiB  (SWISS-4q P0-2 raised from 1 MiB)
```

Twrite emits one extent per call until the writeback-aggregation
layer covers the gap (`v2/src/dirty_buffer/`); the 8 MiB ceiling
matches `STM_FS_RECORDSIZE_MAX` so a 1 GB sequential write produces
~128 extents instead of ~1024. Per-connection buffers heap-allocate
at msize_max â†’ ~16 MiB per connection (req + resp).

### Per-connection caps

```
STM_9P_MAX_WALK        16  (Twalk per-call cap; iterative deep walks must batch)
STM_9P_NAME_MAX       255  (per-component cap; matches Linux NAME_MAX)
STM_9P_MAX_FIDS      4096  (per-connection fid table)
STM_9P_MAX_BINDINGS   128  (per-connection namespace bindings; namespace.tla::MaxBindsPerConn)
STM_9P_NS_PATH_MAX   4096  (per-fid canonical absolute path; matches Linux PATH_MAX)
```

## Public API

```c
stm_status stm_9p_server_create (stm_fs *fs, uint64_t root_dataset,
                                 uid_t auth_uid, gid_t auth_gid,
                                 uint32_t msize_max, stm_9p_server **out);

stm_status stm_9p_server_handle (stm_9p_server *s,
                                 const uint8_t *req, uint32_t req_len,
                                 uint8_t *resp, uint32_t resp_cap,
                                 uint32_t *resp_len);

uint32_t   stm_9p_server_msize  (const stm_9p_server *s);
void       stm_9p_server_destroy(stm_9p_server *s);
```

- `_create` binds a `stm_fs *` (non-owning), a root dataset, and the
  peer's resolved uid/gid. Caller MUST keep `fs` alive for the
  server's lifetime AND tear down the server BEFORE `stm_fs_unmount`.
- `_handle` processes ONE complete 9P request. `req_len` must equal
  the 4-byte LE size prefix. In-band errors (invalid fid, ENOENT,
  EACCES) produce an Rlerror reply and return STM_OK â€” the caller
  writes the reply and continues. Fatal protocol errors (malformed
  frame, oversized msg) return non-OK with `*resp_len == 0`; the
  caller MUST close the connection.

## Implementation

### Server state

Per-instance: `stm_fs *fs` borrowed pointer, negotiated `msize`
(starts at STM_9P_MSIZE_MIN until Tversion lands), per-connection
fid table, per-connection namespace bindings table.

Fid table: open-addressed up to `STM_9P_MAX_FIDS` (4096) entries
keyed by uint32 fid. Each entry caches the inode's identity at bind
time: `(dataset_id, ino, cached_gen, is_open, mode, ns_path)`.

### fid.tla composition

Every Tread / Twrite / Tgetattr / Tsetattr / Tlock / ... handler
calls `verify_fid_fresh` BEFORE forwarding to stm_fs:

```c
verify_fid_fresh(fid):
  cur_gen = stm_fs_inode_gen(fid.dataset_id, fid.ino)
  if (cur_gen != fid.cached_gen) return ESTALE
  if (!stm_fs_inode_alive(fid.dataset_id, fid.ino)) return ESTALE
  return OK
```

This is the runtime gate that closes `fid.tla::IOReject`. Skipping
it on any handler is a confused-deputy hole: client A holds fid F â†’
ino X gen 5 (a private key file). Server unlinks ino X, then a
different client creates a new file at ino X (allocator reused; gen
â†’ 6). Without the gate, A's IO via F still succeeds against the new
file's contents.

Every Twalk / Tattach binding handler snapshots the inode's CURRENT
gen at bind time (`fid.tla::WalkBindsWithCurrentGen`).

### namespace.tla composition

Per-connection bindings table (capped at 128) stores `(prefix â†’ ns_path)`
substitutions. Twalk's per-component loop consults bindings BEFORE
falling through to `stm_fs_lookup`. Tunbind removes; Tbind installs.

UNION_OVER / UNION_UNDER bind modes are accepted on the wire but
return ENOTSUP at v2.0 (deferred to v2.1+ where the spec gets
extended for layered composition). REPLACE mode is what
`namespace.tla` models.

Cross-connection isolation is structural: each server instance owns
its bindings table; there is no global table. namespace.tla's six
buggy variants enumerate the failure modes a reviewer must rule out:
BuggyGlobalBindings, BuggyDetachLeaks, BuggyUnbindCrosstalk,
BuggyLookupCrosstalk, plus two multi-Attach buggies from P9-9P-2c
(`namespace_multi_bind_no_rollback_buggy.cfg` and
`namespace_multi_bind_truncate_only_buggy.cfg`).

### locks.tla composition

Tlock / Tgetlock handlers route to `stm_fs_lock` / `stm_fs_getlock`.
Owner ID derives from the per-conn `auth_uid` âŠ• a per-conn salt so
two clients with the same uid see independent owners. Every fid that
acquired byte-range locks MUST call `stm_fs_release_lock_owner` at
Tclunk + at Tdetach â€” composes against `locks.tla::ReleaseOwner`.

### Wire framing + trust boundaries

Every Txx parser:
- bounds-checks `body_len` BEFORE consuming each field (R71 P1-1 lesson);
- refuses oversized requests (`size > negotiated msize`);
- refuses `n_names > STM_9P_MAX_WALK` and per-name `len > STM_9P_NAME_MAX`.

Every Rxx writer:
- bounds-checks `resp_cap` BEFORE serialising each field;
- clamps `count` fields against BOTH `msize - hdr - 4` AND
  `resp_cap - hdr - 4` so a small response buffer doesn't fail the
  request â€” just emits less.

### Five buggy configurations (audit-reviewer rule-out)

`fid.tla` ships five buggy configs the reviewer of any 9P-server
change MUST trip on the spec before signing off:

1. **Staleness gate skipped** â€” Tread on a stale fid succeeds.
2. **Bind-time gen-snapshot wrong** â€” Twalk binds cached_gen=0.
3. **Tclunk leaks fid** â€” fid stays bound after Tclunk.
4. **Tdetach leaks fids** â€” connection's fid table not cleared.
5. **IOReject fires falsely** â€” fresh fid rejected as stale.

## Spec cross-reference

`v2/specs/fid.tla` pins:

- **`IOSuccess`** â€” fresh fid + alive inode + matching gen â‡’ IO succeeds.
- **`IOReject`** â€” stale gen OR dead inode â‡’ ESTALE.
- **`WalkBindsWithCurrentGen`** â€” bind operation snapshots cur gen.
- **`ClunkClearsBinding`** â€” Tclunk fully clears fid state.

`v2/specs/namespace.tla` pins:

- **`PerConnectionIsolation`** â€” A's binding never visible to B.
- **`BindLookupConsistency`** â€” every Lookup goes through bindings.
- **`UnbindReverts`** â€” Unbind restores original lookup behavior.
- **`MaxBindsPerConn`** â€” refuse the 129th bind with EOVERFLOW.

`v2/specs/locks.tla` pins the byte-range-locking discipline shared
with `stm_fs`.
