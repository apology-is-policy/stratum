# P9.5-PARALLEL-3 — Per-inode `fs->lock` granularity

**Status**: design-first (spec-first applies — TLA+ in `v2/specs/compound_ops_per_inode.tla`).
**Predecessor**: P9.5-PARALLEL-2 (compound-op contract; tip `be81384`).
**Predecessor audit closed**: R132 (compound-op race-class baseline; clean against current code).
**Audit class triggered**: every "Extent write ordering" + "Writeback aggregation" + every public `stm_fs_*` API surface in CLAUDE.md.
**Spec**: `v2/specs/compound_ops_per_inode.tla` (this chunk introduces it; refines `compound_ops.tla`).
**Spec composition**: composes against `compound_ops.tla` (the baseline this refines), `sync.tla`, `writeback.tla`, `snapshot.tla`, `inode.tla` (the (ino, gen) tuple-uniqueness invariant).

## 1. Why this chunk exists

The PARALLEL-2 close documented the /ctl/ multi-subsystem read contract under the **big-fs-lock** model: every public `stm_fs_*` mutating function holds `fs->lock` from entry to exit, and that's the source of the per-subsystem-linearizable + cross-subsystem-eventually-consistent guarantee.

The big-fs-lock is the next concurrency bottleneck:

- Two concurrent `stm_fs_truncate` calls on different inodes serialize even though they touch disjoint state (extent records keyed by `(dataset, ino, offset)`, dirty_buffer keyed by `(dataset, ino)`).
- A long-running `stm_fs_create_snapshot` (which does `fs_flush_all_locked` — a drain of every dirty inode in every dataset, see fs.c:5031) blocks ALL other FS operations for the duration of the drain.
- `stm_fs_copy_file_range` between two independent file pairs serializes despite no shared inodes.

PARALLEL-3 refines `fs->lock` to per-inode granularity for the public APIs that touch a bounded set of inodes (1 to 4), while keeping a `fs->global` lock for ops that span the entire dataset (snapshot create/delete, scrub orchestration, send/recv).

**Crucially, the refinement must preserve PARALLEL-2's contract**:

- Per-subsystem linearizability still holds — each subsystem's internal mutex (allocator, sync, snapshot index, dirent index) is unchanged.
- Cross-subsystem eventual consistency still holds — /ctl/ readers take per-subsystem internal locks one at a time.
- The new failure modes (deadlock from out-of-order lock acquisition, atomicity loss from per-inode lock release mid-op) are formally specified and the impl must rule them out.

### 2.4 Four-inode ops (rename)

`stm_fs_rename` (fs.c:3004) touches up to four inodes:

- `src_parent_ino` (dirent removal)
- `dst_parent_ino` (dirent install)
- `src_ino` (ctime stamp; possibly nlink ops)
- `dst_ino` (if overwrite branch: nlink decrement, cascade-free with R128/R130 wiring)

The overwrite branch is the audit-trigger surface — it's the only path that combines four inodes + extent reclamation.

For these, the lock key is `(dataset_id, ino)` — ascending lex order across the pair.

### 2.6 Dataset-wide ops (keep `fs->global`)

| Function | Why |
|---|---|
| `stm_fs_create_snapshot` | `fs_flush_all_locked` drains every dirty inode in every dataset |
| `stm_fs_delete_snapshot` | walks the freed-paddr list; touches allocator + CAS for every freed extent |
| `stm_fs_create_dataset` / `_destroy_dataset` | dataset table mutation |
| `stm_fs_rollback` | (not yet shipped) rewrites entire dataset state |
| `stm_fs_send` / `_recv` (when /ctl/-exposed) | walks entire snapshot's extent map |
| `stm_fs_commit` | flushes every dirty inode + advances sync gen |

These ops take `fs->global` in EXCLUSIVE mode. Per-inode ops take `fs->global` in SHARED mode, then their target inode locks.

## 3. The refinement strategy

### 3.1 Three-tier locking

| Tier | Lock | What it protects | Acquisition order |
|---|---|---|---|
| 1 | `fs->global` (rwlock) | dataset-wide ops; SH for per-inode ops to read dataset roster | Top of hierarchy |
| 2 | `dataset->root_mu` (mutex; future-noted, not in v1) | dataset-wide state shared by per-inode ops (currently uses fs->lock implicitly) | Below `fs->global`, above per-inode |
| 3 | per-inode mutex | inode metadata + extent records + dirty_buffer entries for that inode | Bottom; ascending (dataset_id, ino) order across multiple inodes |

`fs->global` is held SH (shared) by per-inode ops so a concurrent EX (exclusive) snapshot-create blocks ALL per-inode mutations until the drain finishes, but per-inode ops on different inodes can proceed concurrently.

### 3.2 Per-inode lock placement

The per-inode mutex lives in the **inode cache entry** — `stm_inode_cache_entry::mu` (proposed; the cache itself is in v2/src/inode/inode.c's `stm_inode_index` cache table). The cache is reference-counted; entries are pinned for the duration of any op that uses their lock.

Open question: today the inode cache is opportunistic (lookups go through `stm_inode_get` which materializes from the persistent tree). The refinement either:

- **Option A (recommended)**: Cache entries are MANDATORY for any in-progress op — `stm_inode_pin(idx, ino)` returns a pinned handle that carries the lock; `stm_inode_unpin` releases.
- **Option B**: Lock striping on a fixed table (e.g., 256 mutexes keyed by `ino & 0xff`). Simpler but allows false sharing (two unrelated inodes hashing to the same stripe serialize).

Option A is the chosen direction. The cache infrastructure mostly exists already (`stm_inode_index` has a load/evict path); the refinement adds the `mu` field + the `pin/unpin` API.

### 3.3 Lock acquisition order (deadlock prevention)

For operations touching multiple inodes:

1. Compute the full set of target `(dataset_id, ino)` pairs upfront.
2. Sort ascending by `(dataset_id, ino)`.
3. `stm_inode_pin` each in sorted order; the pin acquires the per-inode mutex.
4. On the success path, `stm_inode_unpin` in any order (mutex release doesn't deadlock).
5. On the failure path, unpin whatever was successfully pinned before the failure point.

This is the canonical Linux/BSD vnode lock-ordering discipline (see Linux `lock_two_inodes` in `fs/inode.c`).

### 3.4 The "create" case (new_ino allocation in the middle)

`stm_fs_create_file` is the awkward case: the new_ino is allocated by the inode allocator AFTER the parent is locked — we don't know the ino number in advance.

Resolution: the inode allocator allocates the ino under the inode-index's internal lock. The new inode is BRAND NEW — no other thread has a reference to it. We pin it WITHOUT acquiring its mutex (since no one else can possibly have it) and proceed. Defense-in-depth: also acquire the mutex on the new pinned handle, knowing it'll succeed instantly.

The "no one else has it" invariant follows from `inode.tla`'s `AllocReused` invariant (R69): until the allocator commits + publishes the new ino, no other code path can resolve to it.

### 3.5 `fs_pre_inode_free_cleanup_locked` + `fs_post_inode_free_reclaim_locked`

These helpers run under the per-inode lock for their target. Both already operate on a single inode's state (dirty_buffer drop + extent truncate + post-reclaim double-commit). The wiring stays per-inode; the only change is which lock is held during the call.

Forward-noted concern: the post-reclaim double-commit calls `stm_sync_commit` which advances `free_gen` — this is per-pool, not per-inode. The double-commit serializes on sync's internal mutex. Two concurrent `stm_fs_unlink` calls on different inodes can each enter their per-inode lock + pre-cleanup, but their post-reclaim commits will serialize. This is correct (sync MUST be serialized), and matches today's behavior under fs->lock — the only difference is the BLOCKING is now at sync->lock instead of fs->lock.

### 3.6 The R128/R130 invariants stay intact

R128 (pre-flush before extent-tree mutation) and R130 (post-reclaim double-commit gated on `cleanup_did_truncate`) are PER-INODE invariants today — they don't depend on holding fs->lock for the whole compound op, only on the SEQUENCING (pre-flush → mutate → post-reclaim).

Under per-inode locking, the sequence stays the same; the lock changes. R128's "any op touching the extent tree MUST pre-flush" rule applies under the per-inode lock.

### 3.7 Cross-inode atomicity (rename specifically)

`stm_fs_rename`'s overwrite branch needs ATOMICITY across the four inodes:

- Drop the old dst_dirent.
- Insert the new dirent pointing at src_ino.
- Remove the src_dirent.
- Decrement src_ino's ctime + dst_ino's nlink + cascade-free dst_ino if nlink→0.

Today this is one big fs->lock hold. Under per-inode: pin all four inodes in ascending order, then do the work. The dirent records (in the global dirent B-tree) are protected by the dirent layer's own internal mutex — concurrent renames touching disjoint dirents don't conflict at the dirent layer.

The PARALLEL-2 "writer holds fs->lock for the full compound op" invariant becomes "writer holds ALL TARGET INODE LOCKS for the full compound op." The `WriterCompoundOpAtomicVsWriter` invariant refines to:

> For each inode `i`, at most one writer holds `i`'s lock at any time.

This is preserved by the per-inode mutex semantics directly.

### 3.8 What `fs->global` does NOT need to protect anymore

Today every public stm_fs_* takes fs->lock. The big-fs-lock implicitly serializes everything. Under per-inode:

- Per-inode ops take `fs->global` SH + their inode lock(s).
- Dataset-wide ops take `fs->global` EX.
- Two per-inode ops on different inodes can proceed concurrently (both hold `fs->global` SH; different inode locks).
- A snapshot create blocks ALL per-inode ops by taking `fs->global` EX (waits for outstanding SH holders to drain).

Subsystem-internal mutexes (allocator, sync, dirent index, snapshot index, scrub) remain. They serialize their own access. Per-inode ops that need to commit MUST take sync's internal mutex, just like today.

## 4. What the spec captures

`v2/specs/compound_ops_per_inode.tla` refines `compound_ops.tla`. New aspects:

- `Inodes` constant (e.g., `{i1, i2, i3}` for a 3-inode model).
- `inode_lock_holder : [Inodes -> Writers \cup {NONE}]` replaces the single `fs_lock_holder`.
- Compound ops declare a target set of inodes upfront (`op_targets[w]`).
- Lock acquisition action `CompoundOpAcquireNext(w, i)` fires only if `i` is the smallest unacquired inode in `op_targets[w]` (deadlock prevention).
- Existing `CompoundOpStep` action now requires the writer to HOLD ALL of its op's target locks.
- Per-subsystem counter mutation is unchanged (still through `CompoundOpStep` with sign + sub_x).

### 4.1 New invariants (cumulative — old ones still apply)

- **`WriterAtomicPerInode`**: for each inode `i`, at most one writer holds the lock for `i`.
- **`LockOrderPreserved`**: for each writer `w`, the inodes it currently holds form a contiguous-from-min prefix of its `op_targets[w]` sorted ascending. (I.e., it cannot hold `{i2, i5}` without also holding the inodes in between if they're in its target set — though it CAN hold `{i2, i5}` if its target set is exactly `{i2, i5}` with nothing between.)
- **`NoDeadlock`**: every reachable state has at least one enabled action OR is a terminal state (all writers have committed up to `MaxOps`).
- **`PerSubsystemLinearizable`** (preserved from `compound_ops.tla`).
- **`NoNegativeCounters`** (preserved from `compound_ops.tla`).
- **`MultiSubReadIsBracketed`** (preserved — documentation of the eventual-consistency contract).

The old `WriterCompoundOpAtomicVsWriter` is REPLACED by `WriterAtomicPerInode` — that's the refinement.

### 4.2 Buggy variants

- **`BuggyOutOfOrderAcquire`**: enables `CompoundOpAcquireOutOfOrder` which can acquire ANY unacquired target (not just the minimum). State space includes deadlock cycles (writer w1 holds i2 wants i3; writer w2 holds i3 wants i2 → both block forever). Trips `NoDeadlock`.
- **`BuggyReleasePerInodeMid`**: enables `WriterReleasesOneInodeMid` which releases ONE of the writer's held inode locks mid-op. A second writer can acquire that inode → two writers concurrently in compound ops for the SAME logical operation set. Trips `WriterAtomicPerInode` if the second writer's op also targets that inode.

A third buggy variant — counter underflow — is inherited from `compound_ops.tla` and need not be re-spelled.

### 4.3 State-space bounds for TLC

```
Writers           = {w1, w2}        \* two writers for the deadlock cycle
Readers           = {r1}
SubKeys           = {a, b}
Inodes            = {i1, i2}        \* two inodes enables 2-cycle deadlock
MaxOps            = 2
MaxStepsPerOp     = 1               \* keep tight to bound state space
MaxReadsPerReader = 1
InitialCounter    = 4
StepDelta         = 2
```

With two inodes + two writers + max-2 op targets, state space is in the low 10s of thousands. The deadlock buggy config explicitly enables the 2-cycle.

Larger configs (3 inodes, 3 writers) explode the state space — not needed at v1.0 since the 2-inode 2-writer cycle is the minimal counterexample for `NoDeadlock`.

### 4.4 What the spec does NOT model

- Specific compound op shapes (rename's 4-inode atomicity, unlink's parent+child). Captured at the C-impl level + tests.
- Actual inode-cache pin/unpin mechanics. The spec models the lock-holder relation abstractly.
- Crash recovery / sync orchestration. Already specified by sync.tla.
- Cross-dataset ordering. Captured by treating the inode ID as opaquely ordered; the impl uses (dataset, ino) lex order.

## 5. The contract that must be preserved

The PARALLEL-2 documented contract carries:

- **Per-subsystem linearizability** — each subsystem's internal mutex still serializes its own state.
- **Cross-subsystem eventual consistency** — /ctl/ readers still take per-subsystem locks one at a time.
- **Writable /ctl/ kinds serialize via fs->global SH + their target inode lock(s)** — admin verbs that touch a single inode (e.g., a hypothetical per-inode property write) take SH + inode lock; admin verbs that touch the whole dataset (snapshot create/delete, scrub trigger) take EX.

PARALLEL-3 ADDS:

- **Per-inode atomicity** — every compound op holds ALL its target inode locks for its full body.
- **Ascending lock-order discipline** — no out-of-order acquisition.
- **Deadlock freedom** — no two writers can block each other in a cycle.

## 10. What this chunk explicitly does NOT do

- **Impl**: no fs.c changes in this commit. Spec phase only.
- **Per-dataset snapshot concurrency**: dataset-wide ops still take `fs->global` EX. Two snapshot creates on different datasets serialize. A future "per-dataset SH" refinement is forward-noted.
- **Crash recovery under per-inode**: sync's three-phase commit is unchanged. The per-inode lock relaxation doesn't change WHEN sync commits, only WHO can race against a sync.
- **Lock-free / RCU readers**: not in scope. PARALLEL-3 stays in the mutex-based model.
- **fcntl byte-range locks**: already per-inode via `stm_lock_table` (P8-POSIX-7d). Orthogonal.

## 11. Why now

PARALLEL-1 made /ctl/ concurrent-accept (`22fa907`).
PARALLEL-2 documented the contract under big-fs-lock (`1a4e174` + `be81384`).
PARALLEL-3 lifts the bottleneck without breaking the contract.

The order is intentional: refining locks without a written contract is how filesystems regress silently. The PARALLEL-2 spec + audit gave us "what today's regime guarantees." PARALLEL-3 proves the per-inode refinement preserves those guarantees. The impl phase has a formal checklist to verify against — not just `ctest`.
