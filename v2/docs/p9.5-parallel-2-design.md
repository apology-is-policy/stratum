# P9.5-PARALLEL-2 — Compound-op race-class audit

**Status**: design-first (spec-first applies — TLA+ in `v2/specs/compound_ops.tla`).
**Predecessor**: P9.5-PARALLEL-1 (concurrent /ctl/ accept; tip `3266599`).
**Predecessor audits closed**: R128 (compound-op pre-flush sites), R130 (perf gate
on post-reclaim double-commit), R131 (concurrent /ctl/ baseline soundness).
**Audit class triggered**: every "Extent write ordering" + "Writeback aggregation"
+ "/ctl/" row in CLAUDE.md.
**Spec**: `v2/specs/compound_ops.tla` (this chunk introduces it).
**Spec composition**: composes against `v2/specs/sync.tla` (commit ordering),
`v2/specs/writeback.tla` (drain discipline), `v2/specs/snapshot.tla` (snap
index invariants).

## 1. Why this chunk exists

The P9.5-PARALLEL-1 close landed concurrent /ctl/ accept (`22fa907`). The
audit-trigger note for it (CLAUDE.md stratumd row clause 4) reads:

> The /ctl/ thread also reads stm_fs (via stm_ctl's getattr_at + materializers)
> concurrently with the FS thread; this is safe under the public stm_fs API's
> thread-safety contract but extends the R94 P2-1 race surface to /ctl/'s
> read paths. **Reviewer of any concurrent-accept upgrade MUST address that
> whole stat-after-mutation class.**

R131 closed at 0 P0/P1 — concurrent /ctl/ is safe AS-IS against the FS-internal
locking discipline. P9.5-PARALLEL-2 is the **soundness-verification chunk**
that turns that audit-against-current-code into a formal model + documented
contract, so future chunks have a written invariant to compose against
rather than re-deriving the analysis each time.

The companion P9.5-PARALLEL-3 chunk lifts fs->lock granularity (per-inode).
That work changes the underlying assumption (no more big-fs-lock), so the
contract this chunk writes down is **the baseline against which PARALLEL-3
proves its refinement**.

## 2. The actual race surface (factual claims, verified)

### 2.1 Inside `stm_fs`: fs->lock is a big-fs-lock, never released mid-op

The Explore audit on `v2/src/fs/fs.c` (5831 lines, ~40 public APIs) verified:

- Every public `stm_fs_*` function that mutates state takes `fs->lock` at
  entry and holds it through the FULL function body.
- Early-unlock paths exist only on **error-rejection branches** (guard
  failures, validation rejects, allocation failures). The success path
  never releases.
- Every compound op with multi-subsystem orchestration runs its sub-steps
  under one continuous `fs->lock` hold:

| Function | fs.c line | Sub-steps (subsystems touched in order) |
|---|---|---|
| `stm_fs_commit` | 701 | dirty_buffer → sync |
| `stm_fs_unlink` / `stm_fs_rmdir` (via `fs_unlink_inode_and_dirent`) | 1947 / 1960 / 1732 | inode → dirent → dirty_buffer (pre-cleanup) → sync (truncate) → inode (unlink) → inode (ctime) → sync (post-reclaim) |
| `stm_fs_unlink_anon` | 1595 | inode → dirty_buffer → sync → inode → sync |
| `stm_fs_create_file` / `stm_fs_mkdir` (via `fs_create_inode_and_link`) | 1364 / 1656 / 1274 | inode → inode → inode → dirent |
| `stm_fs_link` / `stm_fs_link_by_ino` | 2127 / 2253 | inode → dirent → inode → dirent → inode → inode |
| `stm_fs_truncate` | 2507 | dirty_buffer (flush) → inode → sync (write_extent or truncate) → inode |
| `stm_fs_rename` (overwrite branch) | 3004 | inode → dirent → dirent (lookup) → dirty_buffer → sync → dirent (swap or unlink+install) → inode → sync (post-reclaim) |
| `stm_fs_create_snapshot` | 4987 | dirty_buffer (flush_all) → dataset → snapshot |
| `stm_fs_delete_snapshot` | 5094 | snapshot → alloc (free loop) → cas (deref loop) |

In every case **fs->lock is held from the first `pthread_mutex_lock(&fs->lock)`
through the last `pthread_mutex_unlock(&fs->lock)`**. The "compound" in
"compound op" is FS-internal multi-subsystem orchestration; the **race
surface against other FS callers is empty** — fs->lock serializes them.

### 2.2 R128 / R130 pre-flush wiring is in place

Every extent-mutating compound op pre-flushes the dirty_buffer:

- `stm_fs_commit`: `fs_flush_all_locked` (line 713)
- `stm_fs_unlink` / `stm_fs_rmdir`: `fs_pre_inode_free_cleanup_locked` (line 1878)
- `stm_fs_unlink_anon`: `fs_pre_inode_free_cleanup_locked` (line 1632)
- `stm_fs_truncate`: `fs_flush_ino_locked` (line 2523)
- `stm_fs_rename` overwrite: `fs_pre_inode_free_cleanup_locked` (line 3220)
- `stm_fs_create_snapshot`: `fs_flush_all_locked` (line 5031)
- `stm_fs_copy_file_range`, `stm_fs_migrate_to_cold`, `stm_fs_promote_to_hot`:
  pre-flush per R128 P2 close.

The post-reclaim double-commit (`fs_post_inode_free_reclaim_locked`) fires
gated on `cleanup_did_truncate` (R130 P1 perf fix). All in-place under
fs->lock.

### 2.3 The novel surface: /ctl/'s cross-subsystem reads

`/ctl/` materializers reading from multiple subsystems serially WITHOUT
holding fs->lock for the duration of the read are a documented
freshness tradeoff:

- `materialize_pool_metrics_prometheus` (synfs.c:1611): pool (snapshot under
  `stm_pool_lock_shared`, released) → fs (`stm_fs_stats_get` + `stm_fs_dataset_count`,
  each takes fs->lock briefly) → scrub (`stm_scrub_status_get`, takes scrub
  internal). Output is then formatted from the captured snapshots after every
  subsystem lock is released (R110 P3-5 / P3-8 doctrine; see lines 1510-1535).

- `materialize_dataset_properties` (synfs.c:1152): 5 successive
  `stm_fs_effective_dataset_property` calls — each re-takes fs->lock. Comment
  at lines 1163-1168 explicitly documents: *"between them, a concurrent
  set_property could cause us to surface a mixed snapshot. /ctl/ is operator
  state, not a transactional surface; this is documented as a freshness
  tradeoff."*

The /ctl/ readers are eventually-consistent across subsystems. A reader of
`/pools/<uuid>/metrics/prometheus` could see:

1. `pool_data_allocated_blocks = A` (from fs_stats sample at time T1)
2. `pool_datasets_total = N` (from dataset_count at time T2 > T1)

— where between T1 and T2 a concurrent `stm_fs_create_dataset` advanced both
counters. The output shows `A` from T1 and `N` from T2. This is the
documented contract.

### 2.4 What's NOT in scope

- **send / recv (`stm_send_*` / `stm_recv_*`)**: operate at the `stm_sync *`
  layer, not at fs->lock. Their concurrency posture is already specified
  (R39 / R60 stale-paddr documentation in `v2/include/stratum/send_recv.h`)
  with explicit caller-mitigation patterns (snapshot-first or pause-writes).
  /ctl/ does not currently expose send/recv surfaces. Out of scope here;
  re-audit IFF send/recv becomes /ctl/-exposed.

- **scrub orchestration**: scrub state mutates under scrub's own mutex.
  /ctl/ admin verbs (`scrub-trigger`) take fs->lock and dispatch through
  `stm_sync_scrub_*`. /ctl/ readers take scrub's internal mutex. Already
  closed under R103 / R104. No new surface in this chunk.

- **fs->lock granularity itself**: PARALLEL-3 changes this. PARALLEL-2
  documents the contract that PARALLEL-3 must preserve.

## 3. What the spec captures

`v2/specs/compound_ops.tla` models a small abstract system:

- **One writer** holding a virtual fs->lock; performs compound ops that
  span N=3 subsystems (`sub_a`, `sub_b`, `sub_c`).
- **K /ctl/ readers** running concurrently; each reader reads from one or
  more subsystems serially.
- Per-subsystem state is a single integer counter (abstracts allocator
  blocks, dataset count, snapshot count, etc.).
- The writer's compound op transitions counters atomically per sub-step.

### 3.1 Modeled state

```
VARIABLES
    fs_lock_held,        \* writer holds fs->lock
    sub_a, sub_b, sub_c, \* committed counters per subsystem
    ops_committed,       \* monotonic counter of completed compound ops
    reader_obs           \* function: reader_id -> sequence of (sub, value)
```

### 3.2 Actions

- **CompoundOpStart(w)** — writer takes fs->lock.
- **CompoundOpStep(w, sub, delta)** — writer advances `sub_x` by `delta`.
  Multiple steps may fire in a compound op (modeling unlink: dirty_buffer
  drop → sync truncate → inode unlink → sync post-reclaim).
- **CompoundOpCommit(w)** — writer increments `ops_committed`, releases
  fs->lock.
- **ReaderRead(r, sub)** — reader observes `sub_x`'s current value, appends
  `(sub_x, value)` to its observation log. The reader does NOT serialize
  with the writer (this is the /ctl/ subsystem-internal lock posture).

### 3.3 Invariants

- **`PerSubsystemLinearizable`** — every value read for `sub_x` is the
  result of some prefix of `sub_x`-transitioning steps committed so far.
  (Trivially true: the model treats per-subsystem mutations as atomic;
  the C implementation enforces this via per-subsystem internal mutexes.)

- **`NoNegativeCounters`** — `sub_a >= 0 /\ sub_b >= 0 /\ sub_c >= 0` always.
  Verifies that compound-op step ordering never produces transiently-negative
  state (e.g., "blocks freed" exceeding "blocks allocated" at any
  intermediate point).

- **`MultiSubReadIsBracketed`** — for any reader whose observation log
  contains reads from N>1 subsystems, the N values must each be consistent
  with SOME valid history prefix. We do NOT require they be from the SAME
  prefix — that's the eventual-consistency contract.

  Formally: there exists a function `t : reader_obs[r] -> 0..ops_committed`
  mapping each observation to a "committed prefix it could have observed",
  and `t(o_i) <= t(o_j)` need NOT hold for i < j (no monotonicity across
  the multi-sub read).

- **`WriterCompoundOpAtomicVsWriter`** — at any state, AT MOST ONE writer
  holds fs->lock. This is the big-fs-lock invariant; PARALLEL-3 refines it.

### 3.4 Buggy configs

Two configs, each enabling exactly one buggy flag:

- **`compound_ops_writer_releases_lock_mid_buggy.cfg`** —
  `BuggyReleaseLockMid = TRUE` enables `WriterReleasesLockMid(w)`. The
  writer releases fs_lock mid-op (in_flight_op stays > 0). A sibling
  writer's `CompoundOpStart` then fires while the first writer is still
  in flight; both have `InFlight(w_i)` true. Trips
  `WriterCompoundOpAtomicVsWriter`.

- **`compound_ops_negative_counter_buggy.cfg`** —
  `BuggyNegativeCounter = TRUE` enables `CompoundOpStepUnchecked(w, s, sign)`
  which fires `sub_x -= delta` regardless of precondition. Trips
  `NoNegativeCounters` within ~6 states.

A "reader-blocks-writer" buggy config was scoped but dropped: it
captures a liveness concern (writer starvation under unbounded reader
hold) rather than a safety invariant violation, and TLC's safety mode
isn't the right tool. The C-side discipline ("/ctl/ readers MUST NOT
take fs->lock") is documented in the design doc + CLAUDE.md /ctl/ row
clause instead.

### 3.5 Bounds for TLC

```
Writers = {w1, w2}        \* two writers so the buggy-release scenario
                          \* can interleave a second writer's start;
                          \* PARALLEL-3 will lift in spec form.
Readers = {r1}            \* one reader; suffices to exercise the
                          \* per-sub linearizability invariant.
SubKeys = {a, b}          \* two abstract subsystems.
MaxOps = 2                \* up to 2 committed compound ops.
MaxStepsPerOp = 2         \* compound ops have at most 2 steps each
                          \* (negative-counter buggy cfg uses 3 to give
                          \* the underflow a path).
MaxReadsPerReader = 2     \* per-reader cap on observation entries.
InitialCounter = 4        \* enough headroom for sign=-1 + StepDelta=2.
StepDelta = 2             \* per-step counter delta.
```

State space: low tens of thousands. Full check expected <60s on a typical
workstation.

### 3.6 What the spec does NOT model

- The actual subsystem mutex implementation (modeled as atomic per-sub steps).
- Crash recovery (orthogonal — already specced in `sync.tla`).
- Nonce uniqueness (already specced in `metadata_nonce.tla`).
- Specific compound op shapes (unlink's pre-flush, snapshot create's
  drain) — those are specced in their own modules.

This is a **composition spec**: it captures the cross-subsystem contract
that emerges from each individual subsystem spec holding. Future
refinements (per-inode fs->lock in PARALLEL-3) will replay this spec
under finer-grained locking and verify the same invariants.

## 4. The documented contract

Output: a new clause in CLAUDE.md (/ctl/ row) and a paragraph in
`v2/docs/REFERENCE.md` codifying:

> **/ctl/ multi-subsystem read contract**: a /ctl/ reader that consults
> multiple stratum subsystems (pool, fs, scrub, snapshot, alloc, CAS)
> in succession sees **per-subsystem-linearizable but cross-subsystem
> eventually-consistent** values. Each individual subsystem read returns
> a value that is the committed state of that subsystem at some instant
> during the read. The N values composed across subsystems are NOT
> guaranteed to reflect a single global instant.
>
> Operationally: a Prometheus scrape of `/pools/<uuid>/metrics/prometheus`
> may show gauges from slightly different transaction generations. The
> gauges are individually correct; the cross-gauge composition may differ
> from any single-instant snapshot of the daemon. Monitoring tooling
> should filter by `last-N-samples` to detect persistent issues rather
> than treating any single scrape as authoritative.
>
> **Invariant (cross-subsystem)**: no combination of values returned by
> sequential /ctl/ subsystem reads is **impossible** under any valid
> daemon history. There is always SOME sequence of committed compound ops
> that produces a state with each of the read values present at SOME
> point. (Verified by `compound_ops.tla::MultiSubReadIsBracketed`.)
>
> Writable /ctl/ kinds (admin verbs) and FS-side operations both take
> fs->lock and serialize. Two concurrent admin verbs serialize; a /ctl/
> admin verb and a FS-side op serialize. Only /ctl/ READ paths and
> FS-side WRITE paths run concurrently — and read paths take
> per-subsystem internal locks, never fs->lock.

This contract is what PARALLEL-3 must preserve when fs->lock becomes
per-inode (or whatever finer grain).

## 6. Test strategy

### 6.1 Spec verification

TLC against `compound_ops.tla` at the documented bounds. Each cfg's
header carries the expected verdict.

- Main config: green; ≥10K states; all 4 invariants hold.
- 2 buggy configs: each refutes the targeted invariant within ~10 states.

Tooling-gated like #958 (sandbox refuses tla2tools.jar). The cfg
header documentation is enough for a one-line manual run on the user's
TLA+ workflow.

### 6.2 Regression test (`tests/test_ctl_compound_ops.c`)

In-process pthread harness:

1. Create `stm_ctl` + `stm_ctl_conn` (one reader conn).
2. Spawn writer pthread: tight loop of `stm_fs_create_dataset` +
   `stm_fs_create_snapshot` + `stm_fs_delete_snapshot` for 200 iterations.
3. Spawn reader pthread: tight loop of `stm_fs_stats_get` +
   `stm_fs_dataset_count` + `stm_snapshot_iter` for 200 iterations.
4. Both threads exit cleanly. Reader's every individual call returns
   `STM_OK` or `STM_ENOENT` (the stale-snap case from R131).
5. Assertions:
   - No `STM_ECORRUPT` from reader (subsystem integrity preserved).
   - No deadlock (test times out at 30s).
   - Counters from reader never go negative.

Pre-fix variant (revert R128/R130): would either fail with `STM_ECORRUPT`
on a read mid-flight OR deadlock on the missing pre-flush.

### 6.3 No new e2e

The existing `v2/tools/stratum/tests/concurrent_ctl.rs` already exercises
two concurrent /ctl/ readers. Adding FS-side writers to that test would
muddy the wall-time bound it verifies. Keep the C-side regression test
as the cross-class race exerciser.

## 7. Risks + mitigations

| Risk | Mitigation |
|------|------------|
| Spec is too abstract to catch real bugs. | The CompoundOpStep abstraction is the exact shape of what fs.c does (multi-step sub-system updates). The bugs we'd catch (lock-release-mid-op, negative counters, reader-grabs-fs-lock) are the documented classes. PARALLEL-3 will refine to per-inode locks; this spec is the "before" baseline. |
| Documented contract under-constrains future /ctl/ kinds (e.g., a future writable kind that needs cross-subsystem atomicity). | The contract explicitly says "writable kinds take fs->lock and serialize." Future writable kinds requiring cross-subsystem atomicity get a dedicated spec extension (e.g., a future "rollback verb" would extend snapshot.tla + this spec together). |
| TLC unverifiable in-tree (tooling-gated). | Same posture as #958. Cfg header comments document expected verdict. User-side TLA+ workflow or a future-session bootstrap allowance can run TLC. The spec stands as a written invariant either way. |
| Regression test under ASAN takes too long. | Bound iterations at 200 (≈ 1s wall). If ASAN cost exceeds 30s, lower to 50 iterations — still triggers the race class within 1-2 runs. |
| R132 surfaces an unexpected P0/P1 (e.g., a missed pre-flush site). | The chunk pivots — write the fix, extend the spec, add a regression. The Implementation Plan step 5 explicitly accounts for this. |

## 8. What this chunk doesn't address

- **Per-inode fs->lock** (P9.5-PARALLEL-3) — still uses one big fs->lock.
- **Compound-op latency** — the post-reclaim double-commit costs ~200ms
  per inode-free with extents. R130 gates against the no-extents case;
  beyond that, the cost is intrinsic to the R50 P2-1 allocator invariant.
- **send / recv composition with /ctl/** — send/recv operates at sync
  layer, not fs->lock. /ctl/ doesn't expose send/recv. Out of scope.
- **POSIX surface gaps for kernel-9P** (P9.5-POLISH-1).

## 10. Why now

PARALLEL-1 made /ctl/ concurrent-accept; readers can now race FS-side
writers freely. PARALLEL-3 will make fs->lock per-inode; that refinement
needs a written contract for "what the concurrent-/ctl/-vs-concurrent-FS
regime guarantees today." PARALLEL-2 writes that contract down and
verifies it formally, so PARALLEL-3 can prove its locking refinement
preserves it. Without this chunk, PARALLEL-3 has nothing concrete to
preserve — refactoring locks without a contract is a recipe for silently
breaking semantics.
